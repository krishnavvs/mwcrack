# Start

Download the installer `motivewave_setup7.0.23.exe` file from the release page. It's the original file, I just upload it so you can have the version of motivewave that this patch was made to work with.

# How to use

In order to have the "free" license you can either download the already patched files from the release page or you can use the python script to do it yourself. The files needed are in `C:\Program Files (x86)\MotiveWave` and `C:\Program Files (x86)\MotiveWave\bin` or wherever you installed motivewave.

## Other Notes

I don't know if this patch can work on future versions, the current version it works is v7.0.23, if motivewave updates and the patch doesn't work just create an issue and I'll see what I can do, or if you fix it, open a pull request.

# MotiveWave Patch Notes

This directory contains a repeatable patch for the MotiveWave crackme challenge.
The target files are:

- `MotiveWave.jar`, the Java application jar.
- `MotiveWave.exe`, the native Windows launcher.

The repeatable patcher is [patch_wave.py](patch_wave.py).

## Usage

Run the patcher with both challenge files:

```powershell
python .\patch_wave.py .\MotiveWave.jar .\MotiveWave.exe
```

The script:

- Creates `.bak` backups if they do not already exist.
- Refuses to patch unknown class or launcher patterns.
- Treats already-patched methods and strings as a no-op.
- Retargets an already-patched launcher if the jar filename changes.
- Bypasses the native launcher image check that rejects a modified jar.
- Removes stale jar signature artifacts after bytecode edits.
- Recomputes the PE checksum after editing `MotiveWave.exe`.

The patched `MotiveWave.exe` uses the jar path you pass to the script. If you run
the patcher with `.\lib\MotiveWave.jar`, the launcher stays pointed at
`lib\MotiveWave.jar`. If you run it with `.\MotiveWave.jar`, the launcher expects
`MotiveWave.jar` beside the exe. Keep the original `lib`, `javafx`, and `jre`
directories next to the launcher, because the launcher still uses those
directories for dependencies and the JVM.

## License Service Patch

The main activation logic is in `cj/p.class`.

Two response methods have the descriptor:

```text
(Lcj/c;)Lcj/m;
```

They normally build a license response from the activation request/server state.
Both methods are replaced with bytecode that returns a successful `cj/m` object:

```text
cj/m.a:Z = true
cj/m.b:Z = true
cj/m.a:Enums$Edition = ORDER_FLOW
```

The boolean helper:

```text
a(Lcj/c;)Z
```

is replaced with:

```text
iconst_1
ireturn
```

This makes the local activation check resolve as successful without needing a
server response.

## Feature Gate Patch

Feature decisions funnel through:

```text
com/motivewave/platform/common/ah.a(
  Enums$Feature,
  Enums$Edition,
  java.util.List
)Z
```

The method already contains normal positive paths for allowed editions and
feature lists. The final fallback was:

```text
iconst_0
ireturn
```

The patch changes only the final fallback value:

```text
iconst_1
ireturn
```

That keeps the method shape, stack maps, locals, and branch structure intact.

## Startup Gate Patch

The next blocker was not the license response. The app could still enter the
startup wizard/exit path before the console opened.

The relevant method is:

```text
MotiveWave.doStart(Ljavafx/stage/Stage;)V
```

The original early flow checked workspace state and branched into the startup
wizard path. The patch preserves the preceding calls, discards the temporary
workspace list, and jumps to the normal startup path:

```text
pop
nop
nop
goto normal_startup
```

This avoids stack-map problems because the patch preserves the method body size
and reaches an existing valid frame.

## Console Verify Patch

`Console.Q()` performs a late verification check and can terminate the app after
the UI starts. It is replaced with a single return:

```text
return
```

`Console.quit()` is intentionally not patched. An earlier version returned
`false`, which caused the JavaFX close event handler to consume the window close
event. Leaving `Console.quit()` intact preserves the normal `X` button shutdown.

## System.exit Guards

Several guard classes call:

```text
java/lang/System.exit(I)V
```

The patcher resolves the `System.exit` method reference from each class constant
pool and replaces direct bytecode calls with:

```text
pop
nop
nop
```

This consumes the exit status and preserves the original code length.

Patched guard classes:

```text
EdgeProX
MotiveWave
bb/c
bb/j
bb/k
bb/p

com/motivewave/platform/ui/console/dock/ScannerDock

com/motivewave/platform/ui/startup/e
com/motivewave/platform/ui/startup/g
com/motivewave/platform/ui/util/gt
```

`com/motivewave/platform/ui/console/Console` is excluded from this generic exit
patch so the normal close flow can still call `System.exit(0)` after saving.

## Jar Signature Artifacts

The jar contained signature metadata under `META-INF`. After class bytes change,
those signature artifacts are stale. The patcher removes:

```text
META-INF/*.SF
META-INF/*.RSA
META-INF/*.DSA
META-INF/*.EC
```

The manifest is kept.

## EXE Launcher Patch

The native launcher originally references:

```text
lib\MotiveWave.jar
-javaagent:lib\MotiveWave.jar
```

The patcher rewrites the ASCII and UTF-16 embedded launcher strings to the jar
path supplied on the command line. For example, using `.\lib\MotiveWave.jar`
keeps:

```text
lib\MotiveWave.jar
-javaagent:lib\MotiveWave.jar
```

The launcher also computes and compares an image value for the application jar.
After the jar bytecode is patched, that check reaches the dialog:

```text
Unable to load MotiveWave, image is corrupted.
```

The patcher changes that native branch so the launcher continues into the normal
startup path with the patched jar.
